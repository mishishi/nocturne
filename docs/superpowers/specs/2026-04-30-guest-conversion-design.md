# 游客转化设计 spec

> 日期：2026-04-30
> 状态：approved

## 背景

当前游客可以完整走完"记录梦境 → 生成故事"流程，但无法发布到梦墙。游客点击"发布"时才发现需要登录，且登录后的 session 合并机制不完善，导致用户预期落差和流失。

## 目标

- 降低游客到注册用户的转化摩擦
- 保证游客在生成故事后能够顺利发布到梦墙
- 登录后自动合并游客的未完成 session，用户无感知

## 触发节点

游客在 **Story 页面点击"发布到梦墙"** 按钮时触发登录。

- 不在 Questions 流程中途打断，保留用户完成任务的流畅体验
- 用户意图明确（发布），转化率高

## 登录方式

### 微信登录（已有）
现有 WeChat OAuth 流程不变。

### 手机号登录（改造）
- 输入手机号 + 密码
- 后端判断：手机号不存在 → 自动创建账户（register）→ 返回登录成功
- 用户无感知"注册"步骤，体验等同于登录

## Session 合并流程

登录成功后，按以下顺序执行：

1. **草稿自动提交**：如果 Questions 页面有未提交的答案，自动提交到服务器（避免页面跳转导致输入丢失）
2. **静默合并**：调用 `migrateSession` API，将游客 session 迁移到注册账户
3. **成功提示**：toast 显示"检测到您有未完成的梦境，已为您保留"
4. **页面跳转**：自动跳转到 Story 页面
5. **本地清理**：清除 Zustand persist 中存储的游客 session 数据

## 边界情况处理

| 场景 | 处理方式 |
|------|----------|
| 游客 session 为空 | 合并无效果，直接完成登录，不报错 |
| 草稿答案为空 | 自动跳过此题，不触发提交 |
| 合并 API 失败 | toast 提示"保存失败，请重试"，保留本地数据，允许重试 |
| 登录后网络中断 | 最多重试 2 次，失败后提示联系客服 |
| 游客已在其他设备登录 | 以服务器数据为准，本地数据被覆盖 |

## 后端改动

### `/auth/phone/login` 改造

**现状**：手机号不存在时返回错误

**改造后**：
```javascript
// 如果用户不存在，自动注册
if (!user) {
  user = await prisma.user.create({
    data: { phone, password: hash(password) }
  })
}
```

## 前端改动

### 1. Story.tsx

在"发布到梦墙"按钮点击时判断登录状态：
- 已登录：直接调用发布接口
- 未登录：跳转登录页，登录成功后返回 Story 页面

### 2. Login.tsx

手机号登录时调用 `/auth/phone/login`：
- 如果返回"用户不存在"，静默调用 `/auth/register` 后再次登录
- 登录成功后触发 session 合并

### 3. WeChatCallback.tsx

微信登录回调后：
- 触发 `migrateSession` 合并
- 跳转回 Story 页面（而非首页）

### 4. Questions.tsx（可选优化）

在 `handleNext` 和 `handleSkip` 中：
- 检测到登录触发时，先自动提交当前答案
- 防止用户在输入框中打了一半的字丢失

## 数据流

```
游客提问页面
  ↓ 点击"发布"
Story 页面
  ↓ 未登录，跳转
登录页（手机号/微信）
  ↓ 登录成功
WeChatCallback / Login
  ↓ ① 草稿自动提交
  ↓ ② migrateSession
  ↓ ③ 清理本地数据
  ↓ ④ 跳转 Story 页面
Story 页面
  ↓ 重新点击"发布"
发布成功
```

## 成功指标

- 游客→注册用户转化率提升（待埋点）
- 发布按钮点击→完成发布的转化率提升
- 用户反馈"无法发布"的投诉减少

## 非目标（本期不做）

- Questions 页面中途登录的进度恢复 UI（后端已支持，前端暂不单独处理）
- 游客 session 的多端同步
- 登录后选择"合并"或"丢弃"的交互
